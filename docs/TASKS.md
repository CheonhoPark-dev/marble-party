# TASKS — 마블 파티

MVP 캡슐(10줄)
1. 목표: 행사 현장에서 모두가 동시에 참여하는 공정한 제비뽑기 경험을 만든다
2. 페르소나: 오프라인 행사(세미나/파티) 참가자들
3. 핵심 기능(=FEAT-1): 방 생성·QR 입장(빠른 참여 + 대기실 확인)
4. 성공 지표(노스스타): 공정하다고 느낀 비율
5. 입력 지표(1~2개): 불만 제기 건수, 규칙 이해도
6. 비기능 요구(최소 1개): 모바일 웹에서 5초 내 입장 완료(현장 네트워크 기준)
7. Out-of-scope(이번엔 안 함): 룰렛/레이싱 실제 플레이, 장애물 조작, 당첨자 연출
8. Top 리스크 1개: MVP만으로 재미/몰입 검증이 부족함
9. 완화/실험 1개: 짧은 영상/종이 프로토타입으로 공정성·재미 설문
10. 다음 단계(바로 할 행동 1개): 행사 참가자 5명에게 입장 흐름 사용성 테스트

## Milestone 1: FEAT-1 입장 흐름 구축(MVP)
컨텍스트 및 목표: PRD의 “범위(MVP)”와 User Flow의 A~E 노드를 충족
사용자 스토리: “참여자로서, QR을 찍고 5초 안에 입장한다(FEAT-1)”
기술 명세: Database Design의 ROOM/PARTICIPANT, TRD의 임시 저장/권한 모델 준수
인수 조건(Acceptance Criteria):
- 진행자가 방 생성 후 QR을 공유할 수 있다
- 참여자가 5초 내 입장 성공(현장 테스트 기준)
- 대기실에서 인원 수가 실시간 갱신된다
자가 수정 지침:
- PRD 지표(불만/규칙 이해도)에 맞춰 UI가 단순한지 검토
- Design System의 톤/컬러 적용 여부 확인

[x] PRD “범위(MVP)”와 User Flow “입장/대기실” 노드를 참고해 진행자/참여자 화면을 구성한다.
[ ] Database Design의 ROOM/PARTICIPANT 구조로 임시 저장 모델을 만든다.
[ ] TRD의 권한 모델에 맞춰 진행자만 “방 생성/종료”가 가능하도록 제한한다.
[x] Design System에 맞춰 QR 공유 화면과 대기실 화면을 제작한다.
[x] 입장 실패 시 안내 문구/재시도 흐름을 넣는다(공정성 체감 손상 방지).

## Milestone 2: 공정성 체감 보강(측정/가드레일)
컨텍스트 및 목표: 공정성 체감을 정량/정성으로 확인
사용자 스토리: “참여자로서, 공정하다고 느끼고 불만이 생기지 않는다”
기술 명세: PRD 성공 지표, TRD 비기능 요구 반영
인수 조건(Acceptance Criteria):
- 불만 제기 흐름(간단 피드백)이 존재
- 규칙 이해도 1문항 체크 가능
자가 수정 지침:
- 공정성 불만을 유발하는 문구/연출이 없는지 검토

[x] PRD “성공 지표”에 맞춘 간단 설문/피드백 UI를 추가한다.
[x] “공정성 체감 가이드” 문구를 Design System 톤에 맞게 반영한다.
[ ] 결과 화면(임시)에서 과장 연출을 피한다.

## Milestone 3: Post-MVP 확장(FEAT-2/FEAT-3)
컨텍스트 및 목표: 룰렛/레이싱/장애물 조작으로 재미 강화
사용자 스토리: “참여자로서, 버튼 조작으로 게임에 기여한다(FEAT-3)”
기술 명세: TRD 실시간 통신 요구, Database Design의 GAME_SESSION/OBSTACLE_ACTION 사용
인수 조건(Acceptance Criteria):
- 장애물 조작이 “막아버리기”로 느껴지지 않음
- 진행자가 게임을 시작/종료 가능
자가 수정 지침:
- 공정성 불만 발생 시 즉시 재시작 플로우 검토

[ ] 장애물 카탈로그(기본/확장)를 PRD 기준으로 정의한다.
[ ] FEAT-2/3 최소 프로토타입을 만들고 소규모 행사에서 테스트한다.
[ ] 불공정 체감 여부를 관측하고 룰을 수정한다.
