# TRD — 마블 파티

MVP 캡슐(10줄)
1. 목표: 행사 현장에서 모두가 동시에 참여하는 공정한 제비뽑기 경험을 만든다
2. 페르소나: 오프라인 행사(세미나/파티) 참가자들
3. 핵심 기능(=FEAT-1): 방 생성·QR 입장(빠른 참여 + 대기실 확인)
4. 성공 지표(노스스타): 공정하다고 느낀 비율
5. 입력 지표(1~2개): 불만 제기 건수, 규칙 이해도
6. 비기능 요구(최소 1개): 모바일 웹에서 5초 내 입장 완료(현장 네트워크 기준)
7. Out-of-scope(이번엔 안 함): 룰렛/레이싱 실제 플레이, 장애물 조작, 당첨자 연출
8. Top 리스크 1개: MVP만으로 재미/몰입 검증이 부족함
9. 완화/실험 1개: 짧은 영상/종이 프로토타입으로 공정성·재미 설문
10. 다음 단계(바로 할 행동 1개): 행사 참가자 5명에게 입장 흐름 사용성 테스트

## 시스템 아키텍처(고수준)
- 진행자 화면(프로젝터용)과 참여자 화면(모바일 웹)을 분리한다.
- 세션 관리 서버가 방 생성/입장/대기 상태를 관리한다.
- 실시간 통신 채널로 “입장/준비” 상태를 즉시 동기화한다.
- 임시 저장소로 세션 상태를 잠시 보관하고, 행사 종료 시 삭제한다.

## 비기능 요구
- 성능: 200명 동시 접속에서 입장 시간 5초 이내.
- 신뢰성: 재접속 시 상태 복구(대기실/준비 상태).
- 공정성 체감: 장애물 조작이 “막아버리기”로 느껴지지 않게 제한.
- 접근성: 큰 글자, 명확한 버튼, 포커스 표시.

## 데이터베이스 요구
- 최소 수집: 실명/연락처 저장 금지.
- 임시 저장: 세션 종료 후 자동 삭제(TTL).
- 통계: 익명 집계만 유지(불만/이해도).

## 외부 API 연동 명세
- 없음. QR은 단순 접속용 링크 생성에만 사용.

## 권장 기술 스택(이유 포함, 벤더 락인 리스크 주석)
- 클라이언트: 설치 없는 웹 앱(현장 참여 장벽 최소화).
- 실시간 통신: 표준 기반 실시간 채널(WebSocket 또는 SSE).
- 서버: 실시간 처리가 쉬운 범용 런타임(Node.js/Go 등).
- 세션 저장: 인메모리 + TTL 저장소(짧은 세션에 적합).
- 벤더 락인 리스크: 특정 플랫폼 전용 실시간 서비스에 의존하면 비용/이전 부담 증가 → 표준 프로토콜 우선.

## 접근제어·권한 모델(역할·정책)
- 역할: 진행자(Host), 참여자(Participant)
- 정책:
  - 진행자: 방 생성/시작/종료, 설정 변경
  - 참여자: 입장/준비 표시, (추후) 장애물 조작

## 데이터 생명주기(최소수집/보존/삭제/익명화 경로)
- 생성: 방 생성 시 room/participant 레코드 생성
- 사용: 대기실에서 실시간 상태 갱신
- 보관: 세션 종료 후 즉시 삭제(TTL)
- 익명화: 통계는 집계 형태로만 보관
